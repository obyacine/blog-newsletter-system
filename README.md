# Blog & Newsletter System —

A FastAPI project combining two systems: blog management and newsletter management, with JWT-based admin authentication.

---

## Tech Stack

- **FastAPI** — API framework
- **SQLAlchemy** — ORM, to talk to the database in Python instead of raw SQL
- **SQLite** — local database (`.db` file)
- **Pydantic** — input/output data validation
- **python-jose** — creating and verifying JWT tokens
- **passlib (bcrypt)** — password hashing
- **python-dotenv** — reading environment variables (`.env`)

### 1. database.py — the building blocks

- **`engine`** → the physical connection to the database (knows where and how to connect)
- **`Base`** → the parent class that all models inherit from
- **`SessionLocal`** → a "factory" that creates sessions
- **`Session`** → a temporary conversation with the database, opened then closed
- **`get_db()`** → a function that opens a session, hands it to the route via `yield`, then closes it in the `finally` block

### 2. Password Hashing

- `pwd_context.hash(password)` → turns `"password123"` into something like `$2b$12$...`
- The **salt** is a random number added before hashing, automatically generated by bcrypt. It ensures that identical passwords produce different hashes → protects against rainbow tables.
- `pwd_context.verify(plain_password, stored_hash)` → compares and returns `True`/`False`.

### 3. JWT (JSON Web Token)

A token = `header.payload.signature`
- **header** → the algorithm (HS256)
- **payload** → the data (e.g. `{"sub": "admin@email.com", "exp": ...}`)
- **signature** → generated with `SECRET_KEY`, guarantees the token hasn't been tampered with

**HS256** = HMAC + SHA-256 → a signing mechanism that combines the data and the secret key to produce a signature that can't be forged without knowing the key.

Full flow:
1. Admin sends email + password
2. Verified with `verify_password`
3. If OK → `create_access_token({"sub": admin.email})` generates the token
4. Admin sends this token back on every request (`Authorization: Bearer ...`)
5. `get_current_admin` decodes the token, looks up the admin in the database, and protects the route

### The Salting Function — in detail

Hashing follows the principle `f(x) = y`: a one-way function, impossible to reverse to retrieve `x` from `y`.

**Problem without a salt**
If two users have the same password `x`, they'll have the same hash `y`. An attacker can precompute a table of known hashes (rainbow table) and instantly recover the original password.

**The solution: the salt**
A unique random number `z` (the salt) is added before hashing:

f(x + z1) = y1
f(x + z2) = y2

Same password `x`, two different salts → two completely different hashes. The rainbow table becomes useless.

**In this project**
`bcrypt` (via `passlib`) handles all of this automatically:
- generates a random salt for each `hash_password()` call
- combines password + salt
- stores the salt directly inside the final hash

So `pwd_context.verify(password, hash)` can retrieve the salt that was used and redo the calculation to compare — without needing to store it separately.

### 4. Depends — dependency injection

`Depends(...)` tells FastAPI: "before running this route, run this function and give me its result".

Uses in this project:
- `Depends(get_db)` → injects a database session
- `Depends(get_current_admin)` → verifies the JWT token and injects the logged-in admin

### 5. CRUD with SQLAlchemy

```python
# Create
new_obj = Article(title=..., slug=..., ...)
db.add(new_obj)
db.commit()
db.refresh(new_obj)   # retrieves the generated id

# Read
db.query(Article).filter(Article.slug == slug).first()   # single result
db.query(Article).filter(Article.is_published == True).all()  # list

# Update
article.title = new_title
db.commit()

# Delete
db.delete(article)
db.commit()
```

### 6. Why no async/await here

Classic SQLAlchemy is **synchronous**. Mixing `async def` with a synchronous ORM would block FastAPI's event loop. For true async, you'd need async SQLAlchemy or Tortoise ORM — not necessary for this project.

### 7. .env and security

- `.env` contains the secrets: `DATABASE_URL`, `SECRET_KEY`, `ALGORITHM`, `ACCESS_TOKEN_EXPIRE_MINUTES`
- `python-dotenv` (`load_dotenv()`) loads these variables into `os.environ`
- `.env` is in `.gitignore` → **never** pushed to GitHub

## API Routes

### Authentication
| Method | Route | Description |
|---|---|---|
| POST | `/auth/login` | Admin login → returns a JWT |

### Blog — public
| Method | Route | Description |
|---|---|---|
| GET | `/articles` | List of published articles |
| GET | `/articles/{slug}` | Article detail |

### Blog — admin (JWT required)
| Method | Route | Description |
|---|---|---|
| POST | `/admin/article` | Create an article |
| PUT | `/admin/articles/{id}` | Update an article |
| DELETE | `/admin/articles/{id}` | Delete an article |
| POST | `/admin/articles/{id}/publish` | Publish / unpublish |

### Newsletter — public
| Method | Route | Description |
|---|---|---|
| POST | `/subscribe` | Subscribe |
| GET | `/unsubscribe/{email}` | Unsubscribe |

### Newsletter — admin (JWT required)
| Method | Route | Description |
|---|---|---|
| GET | `/admin/subscribers` | List of active subscribers |
| POST | `/admin/newsletter/send` | Send a newsletter |
| GET | `/admin/newsletter/history` | Send history |
